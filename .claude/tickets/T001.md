# T001 — Monorepo Scaffold

## Why this ticket exists
Without a consistent folder layout and empty-but-compiling service skeletons, every subsequent task has nowhere to land. If service folders don't exist, no one can add a route, a migration, or a Kafka consumer — the entire build chain starts here. This ticket also establishes the polyglot reality of the project: .NET and Java services coexist in the same repo, so shared tooling files (.editorconfig, .gitignore) need to handle both ecosystems from day one.

All development code lives under `src/` to keep it clearly separated from Claude tooling (`.claude/`, `CLAUDE.md`, `SPECS.md`, `PROGRESS.md`).

- `/src/contracts/kafka/` — empty directory for JSON schema files (T002 fills it)
- `/src/contracts/openapi/` — empty directory for OpenAPI YAML files (T005 fills it)
- `/src/services/api-gateway` — `dotnet new web -n api-gateway` (.NET 9)

## Technical specification

Create the top-level folder structure:

```
/ShareConnectSave
  /src                        <- all development code lives here
    /contracts
      /kafka                  <- JSON schema files per topic (T002)
      /openapi                <- OpenAPI YAML per service (T005)
    /services
      /api-gateway            (.NET 9)
      /user-service           (.NET 9)
      /chat-service           (.NET 9)
      /notification-service   (.NET 9)
      /discovery-service      (Java 21 / Spring Boot 3)
      /connection-service     (Java 21 / Spring Boot 3)
      /rating-service         (Java 21 / Spring Boot 3)
      /report-service         (Java 21 / Spring Boot 3)
      /admin-service          (Java 21 / Spring Boot 3)
    /frontend                 (Angular 17)
    /infra
      docker-compose.yml
      docker-compose.override.yml
  .editorconfig               <- repo root (applies to all files in repo)
  .gitignore                  <- repo root
  .claude/                    <- Claude Code tooling (tickets, skills, etc.)
  CLAUDE.md
  SPECS.md
  REQUIREMENTS.md
```

Initialize each service as an empty project:
- .NET services: `dotnet new web -n <service-name> -o src/services/<service-name>`
- Java services: Spring Initializr with Spring Web, Spring Data JPA, Spring Kafka, Validation, Actuator, Lombok — unzip into `src/services/<service-name>`
- Angular: `ng new frontend --routing --style=css` run from inside `src/`

## Patterns demonstrated
**Monorepo** — all services share a single git repository, enabling atomic commits across service boundaries and a single source of truth for contracts.
**Database per Service** — separate service folders signal the architectural boundary: each service owns its own code and, later, its own schema.
**Polyglot persistence setup** — .NET and Java services coexist from the start, demonstrating that microservices can use the best runtime for each job without coupling.
**Convention over configuration** — `.editorconfig` enforces a single coding style across teams and languages without relying on individual IDE settings.
**Separation of concerns (repo layout)** — `src/` holds all runnable code; the repo root holds tooling, specs, and docs. A developer knows exactly where to look for each concern.

- Angular: use module-based style (NOT standalone). The project uses NgModule — no standalone components. Run `ng new` from inside the `src/` directory so the project lands at `src/frontend/`.
- docker-compose.yml stubs use `image: shareconnectsave/<service-name>:dev` with a `build: context` placeholder. Do NOT attempt to build images — that is T004's job.
- Do NOT add business logic, routes, or database config — this ticket is scaffold only.
- The `.gitignore` must include: `bin/`, `obj/`, `target/`, `node_modules/`, `.angular/`, `*.user`, `*.suo`, `.env`, `*.log`.
