# T003 — Shared Error Envelope Contract

## Why this ticket exists
The platform has four .NET services and five Java services, all sitting behind a single API Gateway. If each service invents its own error shape — some use `error`, some use `message`, some include `traceId`, some don't — the Angular frontend has to handle 9 different error formats and the gateway can't produce meaningful traces. This ticket defines one error envelope that every service speaks regardless of language, and delivers the shared Java library that enforces it across all Spring Boot services without copy-pasting handler code.

## What to build
- `/contracts/openapi/error-envelope.yaml` — OpenAPI 3.1 component schema for `ErrorResponse` (fields: `code`, `message`, `traceId`)
- `/services/user-service/src/Contracts/ErrorResponse.cs` — C# record matching the same JSON shape (model for all .NET services)
- `/services/shared-java-lib/` — Maven module:
  - `pom.xml` — group: `com.shareconnectsave`, artifact: `shared-java-lib`
  - `src/main/java/com/shareconnectsave/shared/ErrorResponse.java` — Java record/POJO with `code`, `message`, `traceId`
  - `src/main/java/com/shareconnectsave/shared/GlobalExceptionHandler.java` — `@ControllerAdvice` with two handlers

## Technical specification

Define the shared HTTP error response shape used by all services.

1. Create `/contracts/openapi/error-envelope.yaml` — OpenAPI component schema for `ErrorResponse`.
2. Create `/services/user-service/src/Contracts/ErrorResponse.cs` as a C# record (copy this pattern to all .NET services in their respective tasks).
3. Create a shared Java library at `/services/shared-java-lib/` — a minimal Maven module with `ErrorResponse.java` and `GlobalExceptionHandler.java` (`@ControllerAdvice`). All Java services will depend on this module.

## Acceptance criteria
- [ ] `ErrorResponse` has fields: `code` (string), `message` (string), `traceId` (string)
- [ ] Java `GlobalExceptionHandler` maps `MethodArgumentNotValidException` → 400 with code `VALIDATION_ERROR`
- [ ] Java `GlobalExceptionHandler` maps unhandled `Exception` → 500 with code `INTERNAL_ERROR`
- [ ] C# record matches same JSON shape

## Patterns demonstrated
**Single Responsibility (SOLID-S)** — `GlobalExceptionHandler` does one thing: translate exceptions into the standard envelope. No business logic, no DB calls.
**Don't Repeat Yourself via shared library** — the Java handler lives in `shared-java-lib` once; all five Java services declare it as a Maven dependency rather than duplicating the class.
**Polyglot contract** — the OpenAPI YAML is the language-neutral source of truth; C# and Java implementations are derived from it, not the other way around.
**Distributed Tracing integration** — `traceId` in the error envelope means every error response carries the OpenTelemetry trace ID, so a client error in the frontend can be correlated directly to a Jaeger trace.
**Dependency Inversion (SOLID-D)** — .NET services use `IProblemDetailsService` (abstraction), not a concrete handler wired per-service.

## Agent implementation notes
- Key files to create:
  - `/contracts/openapi/error-envelope.yaml`
  - `/services/user-service/src/Contracts/ErrorResponse.cs`
  - `/services/shared-java-lib/pom.xml`
  - `/services/shared-java-lib/src/main/java/com/shareconnectsave/shared/ErrorResponse.java`
  - `/services/shared-java-lib/src/main/java/com/shareconnectsave/shared/GlobalExceptionHandler.java`
- C# record must be annotated with `[JsonPropertyName]` to guarantee snake_case JSON keys (`code`, `message`, `traceId`) regardless of serializer settings
- .NET: wire the error envelope via `app.UseExceptionHandler` + a custom handler delegate in `Program.cs` of user-service — this is the pattern that all other .NET services will copy in their respective tasks
- Java `GlobalExceptionHandler`: annotate with `@ControllerAdvice` and `@RestController`. The `traceId` must be read from `MDC.get("traceId")` (populated by OpenTelemetry) — do not generate a random UUID here
- `shared-java-lib` is a plain Maven module — no Spring Boot parent, no `@SpringBootApplication`. It only declares `spring-web` and `spring-boot-starter-validation` as dependencies
- Do NOT add `shared-java-lib` to the Java services' `pom.xml` files yet — that is each service's own task. Just create the library module itself.
- Do NOT copy `ErrorResponse.cs` to the other .NET services — they each do that in their own phase tickets.
- The `error-envelope.yaml` must be a reusable `components/schemas` entry, not a standalone file — it will be `$ref`'d by T005's OpenAPI specs.
