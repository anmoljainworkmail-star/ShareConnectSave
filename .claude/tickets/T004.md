# T004 — Docker Compose Infrastructure Services

## Why this ticket exists
Every microservice in this platform depends on at least one external system: SQL Server for relational data, MongoDB for chat messages, Kafka for async events, and Redis for caching and SignalR scaling. Without a working Docker Compose file, no developer can run any service locally, no integration test can connect to a real database, and no later phase can prove it works end to end. This ticket gives the whole team a one-command local environment.

## What to build
- `/infra/docker-compose.yml` with these services:
  - `sqlserver` — SQL Server 2022 (`mcr.microsoft.com/mssql/server:2022-latest`), port 1433, healthcheck via `sqlcmd`
  - `mongodb` — MongoDB 7 (`mongo:7`), port 27017, healthcheck via `mongosh --eval "db.adminCommand('ping')"`
  - `zookeeper` — Bitnami Zookeeper (`bitnami/zookeeper:latest`), internal only
  - `kafka` — Bitnami Kafka (`bitnami/kafka:latest`), port 9092, depends on zookeeper, healthcheck via `kafka-topics.sh`
  - `redis` — Redis 7 (`redis:7-alpine`), port 6379, healthcheck via `redis-cli ping`
  - `kafka-ui` — Kafka UI (`provectuslabs/kafka-ui:latest`), port 9000, depends on kafka
- `/infra/docker-compose.override.yml` — dev overrides: named volumes for hot reload, `.env` file reference
- `/infra/.env.example` — documents required environment variables (SA password, etc.)

## Technical specification

Define all infrastructure dependencies in `docker-compose.yml`:

- **SQL Server 2022** — single instance, each service gets its own database (not schema) created on first run
- **MongoDB 7** — single instance, databases: `chat_db`, `report_db`
- **Kafka (Bitnami)** + Zookeeper — single broker for dev
- **Redis 7** — used by Discovery Service (cache) and SignalR backplane
- **Kafka UI** (provectuslabs/kafka-ui) — dev convenience, port 9000

All infra services have healthchecks. Application service stubs reference infra by container name (see REQUIREMENTS.md service URL table).

## Acceptance criteria
- [ ] `docker compose up` starts all infra without errors
- [ ] SQL Server reachable at `localhost:1433`
- [ ] MongoDB reachable at `localhost:27017`
- [ ] Kafka broker reachable at `localhost:9092`
- [ ] Redis reachable at `localhost:6379`
- [ ] Kafka UI accessible at `http://localhost:9000`

## Patterns demonstrated
**Infrastructure as Code** — the entire dev environment is defined in a version-controlled YAML file; any developer can reproduce it with one command.
**Database per Service** — SQL Server runs as a single container but each service will create its own database (`user_db`, `discovery_db`, etc.), enforcing the architectural boundary at the data layer.
**Healthchecks as dependency gates** — `depends_on: condition: service_healthy` ensures Kafka does not start before Zookeeper is ready, and application services won't start before their databases are accepting connections.
**Named networks** — all containers share a named Docker network (`shareconnectsave-net`), so services resolve each other by container name (e.g., `kafka:9092`) rather than IP address.
**Environment separation** — `docker-compose.override.yml` carries dev-only settings (volume mounts, debug ports); production would use a separate override without any of these.

## Agent implementation notes
- Key files to create:
  - `/infra/docker-compose.yml`
  - `/infra/docker-compose.override.yml`
  - `/infra/.env.example`
- Do NOT use the `version:` key at the top — it is deprecated in Compose v2
- SQL Server: set `ACCEPT_EULA=Y` and `MSSQL_SA_PASSWORD` from `.env`. The SA password must meet SQL Server complexity requirements (uppercase, number, symbol). Use `SA_PASSWORD` in `.env.example` with a placeholder value.
- SQL Server database creation: use an init script mounted at startup — SQL Server does NOT support `/docker-entrypoint-initdb.d/`. Instead mount a `.sql` file and run it via a `command` override using `sqlcmd`. Create databases: `user_db`, `discovery_db`, `connection_db`, `chat_db` (empty — MongoDB owns messages), `rating_db`, `notification_db`, `report_db`, `admin_db`.
- Bitnami Kafka: set `KAFKA_CFG_ADVERTISED_LISTENERS=PLAINTEXT://localhost:9092` so the broker is reachable from the host machine for local testing. Also set `KAFKA_CFG_LISTENERS=PLAINTEXT://:9092`.
- Do NOT create Kafka topics here — topics are documented in T002 and created by each service on startup (Spring Kafka auto-creates; .NET creates via `IHostedService`).
- Redis: no auth in dev — that is fine for local only.
- Kafka UI: set `KAFKA_CLUSTERS_0_NAME=local` and `KAFKA_CLUSTERS_0_BOOTSTRAPSERVERS=kafka:9092`.
- Application service stubs (api-gateway, user-service, etc.) may be omitted from this compose file — they are added in their respective phase tickets. This ticket is infra-only.
- `.env.example` must NOT contain real passwords — only placeholder values and comments explaining each variable.
