# T005 — OpenAPI Specs (All Services)

## Why this ticket exists
Every later phase has a developer agent implementing a service. Without a written API contract, each agent guesses the endpoint paths, request bodies, and response shapes — and they will all guess differently. This ticket gives every agent a single source of truth to implement against, the same way a real team uses a design-first API spec before writing any code. It also gives the Angular team the spec they need to generate TypeScript clients in Phase 11.

## What to build
- `/contracts/openapi/user-service.yaml` — auth, profile, OTP, identity verification, block
- `/contracts/openapi/discovery-service.yaml` — scan start/stop, nearby users, BLE token resolve, women-only toggle
- `/contracts/openapi/connection-service.yaml` — send request, list pending, accept, decline, expire
- `/contracts/openapi/chat-service.yaml` — send message, mark Met Successfully, close chat
- `/contracts/openapi/rating-service.yaml` — submit rating, get trust score
- `/contracts/openapi/notification-service.yaml` — internal only; document FCM push + SignalR fan-out trigger endpoints
- `/contracts/openapi/report-service.yaml` — file report, list reports (admin)
- `/contracts/openapi/admin-service.yaml` — review queue, override trust score, suspend/reinstate, view report history

Each file must `$ref` `./error-envelope.yaml` for all error responses.

## Technical specification

Author one OpenAPI 3.1 YAML file per service. These are the contracts subagents implement against.

Files:
- `user-service.yaml`
- `discovery-service.yaml`
- `connection-service.yaml`
- `chat-service.yaml`
- `rating-service.yaml`
- `notification-service.yaml`
- `report-service.yaml`
- `admin-service.yaml`

Each file must cover all endpoints from REQUIREMENTS.md for that service. Include:
- Request/response schemas (reference `error-envelope.yaml` for errors)
- Auth note: "JWT validated at gateway; services receive X-User-Id header"
- All HTTP status codes (200/201/400/401/403/404/409/500)

## Acceptance criteria
- [ ] All 8 YAML files exist and validate against OpenAPI 3.1 spec
- [ ] Every endpoint from REQUIREMENTS.md is represented
- [ ] Error responses reference the shared `ErrorResponse` component

## Patterns demonstrated
**API-first / Contract-first design** — the spec exists before the code; implementation conforms to the contract, not the other way around.
**Single source of truth** — eight specs in one place; Angular, gateway routing, and service implementations all derive from the same document.
**Open/Closed Principle (SOLID-O)** — adding a new endpoint means extending the spec file, not modifying existing endpoint definitions.
**Interface Segregation (SOLID-I)** — one spec file per service means no service spec pollutes another; a consumer only reads what it depends on.
**Security by design** — every endpoint documents the auth model (`X-User-Id` header injection from gateway) and the 401/403 responses, making the security contract explicit before any code runs.

## Agent implementation notes
- Key files to create: 8 YAML files in `/contracts/openapi/`
- Use OpenAPI 3.1.0 (`openapi: 3.1.0`), not 3.0.x — 3.1 is JSON Schema compatible
- Global `info.description` on each file: "JWT validated at API Gateway (YARP). Services receive X-User-Id (uuid), X-User-Role (user|admin), X-User-Gender (female|male|unspecified) headers. Never re-validate the JWT."
- All error responses must use: `$ref: './error-envelope.yaml#/components/schemas/ErrorResponse'`
- Endpoints to cover per service (derive request/response schemas from REQUIREMENTS.md functional requirements):
  - **user-service**: `POST /auth/google`, `POST /auth/otp/send`, `POST /auth/otp/verify`, `GET /profile`, `PUT /profile`, `POST /profile/photo`, `POST /identity/verify`, `GET /identity/status`, `POST /blocks/{userId}`, `DELETE /blocks/{userId}`, `GET /blocks`
  - **discovery-service**: `POST /scan/start`, `DELETE /scan/stop`, `GET /scan/nearby`, `POST /ble/resolve`, `PUT /settings/women-only`
  - **connection-service**: `POST /connections`, `GET /connections/pending`, `PUT /connections/{id}/accept`, `PUT /connections/{id}/decline`, `GET /connections/{id}`
  - **chat-service**: `GET /chats/{connectionId}`, `POST /chats/{connectionId}/messages`, `POST /chats/{connectionId}/met`, `DELETE /chats/{connectionId}` (admin/timeout close)
  - **rating-service**: `POST /ratings`, `GET /ratings/trust-score`, `GET /ratings/trust-score/{userId}` (admin)
  - **notification-service**: `POST /internal/notify` (gateway-to-service internal trigger — document as internal only)
  - **report-service**: `POST /reports`, `GET /reports` (admin), `GET /reports/{id}` (admin)
  - **admin-service**: `GET /admin/queue`, `PUT /admin/users/{id}/trust-score`, `PUT /admin/users/{id}/suspend`, `PUT /admin/users/{id}/reinstate`, `GET /admin/users/{id}/reports`
- Chat has NO `GET /messages` history endpoint — that is by design (messages are ephemeral). Do not add one.
- Women-only toggle is a settings endpoint, not a discovery query param — the server applies it as a stored filter predicate.
- Do NOT add authentication schemes to the OpenAPI `securitySchemes` — JWT is consumed at the gateway and never reaches services directly.
